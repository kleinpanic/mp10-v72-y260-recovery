# Monoprice MP10 V72 Y-Boundary Firmware Patch

Unofficial firmware patch for full-size Monoprice MP10 / Malyan MA10-class printers where the stock v72 calibration path drives the Y axis farther forward than the machine can physically travel.

This repo is intentionally scoped: one patched firmware image, reproducible tooling, the patch manifest, checksums, the available vendor C/C++ source reference, generated binary-analysis output, and license/notice files.

## What This Changes

The official full-size MP10 v72 firmware can target the high-Y calibration/UBL row near `Y290`. On the tested printer, the moving bed hit the front hard limit around `Y260..Y270`, causing grinding/slapping during calibration.

This patch keeps the v72 firmware family and changes the Y boundary path to `Y260`.

- Base image: official full-size MP10 v72 `update.bin`
- Base SHA-256: `0f55858473446b5987b6ff4f010126adac1023686595c85b4c45c00de5dbd55a`
- Patched SHA-256: `51e61467e79c5cfd9f59300a8665577da1bed0d14209e6dc717ff7c3f374a7f5`
- Patch method: validated binary patch, not a fake source rebuild
- Build/release artifact: `firmware/custom_update.bin`
- SD-card bootloader filename: `update.bin`

The patch edits five 32-bit float words documented in [firmware/manifest.md](firmware/manifest.md):

- Y soft endstop values: `300.0 -> 260.0`
- UBL high-Y mesh coordinate: `290.0 -> 260.0`
- Shared UBL max clamp constants: `290.0 -> 260.0`

## Source And Decompilation

The MP10 firmware is an STM32 ARM firmware image that was originally built from C/C++ firmware code. This repository now includes the available Monoprice-published MP10-family C/C++ source archive under [src/vendor-reference](src/vendor-reference/).

Source provenance:

- Upstream repository: `git@github.com:Monoprice3DP/MiniV2DeltaSourceCode.git`
- Upstream commit captured locally: `e800c654117d51b0070820aad61132eb60b2ffeb`
- Imported archive: `MM32-F401-MARLIN-V2-HAL-5844f073761aac2619543da0502dfc4524c2a1d5.zip`
- Imported archive SHA-256: `157338cd85d7858ad13ffcd51cb04d45c533448688c1258ec4aebaec24e01ceb`
- Upstream README description: "The source code for the mini v1, v2, MP10, MP10 Mini and Mini Delta."

The source import includes the upstream README and GPLv3 license, the original ZIP archive under `src/vendor-reference/Monoprice3DP-MiniV2DeltaSourceCode/archives/`, and the extracted C/C++ tree under `src/vendor-reference/Monoprice3DP-MiniV2DeltaSourceCode/extracted/`.

The imported vendor source archive is real C/C++ source. Its main `Marlin/Configuration.h` contains a full-size MA10 block with `X_BED_SIZE 300`, `Y_BED_SIZE 300`, and `Z_MAX_POS 400`, plus other shared/default/example configurations for smaller machines. The tree is still a shared MP10-family source drop rather than a proven exact source-build input for the official full-size v72 binary: `Marlin/inc/version.h` reports build `66`, while the official artifact patched here is v72. For that reason, the official full-size MP10 v72 `update.bin` package remains the authority for the shipped firmware artifact, and this repository still does not claim a clean source rebuild.

The Python files are tools, not firmware source:

- `tools/create_update_bin.py` validates the official v72 binary hash and applies the documented Y260 binary patch.
- `tools/decompile_official_v72.py` runs Ghidra against the official binary and writes generated analysis under `src/official-v72/`.
- `src/vendor-reference/` is the imported original vendor C/C++ source reference.
- `src/official-v72/decompiled.c` is C-like pseudocode generated by Ghidra, not original vendor C/C++.

## Use This Only If

- Your printer is the full-size MP10 / MA10-class machine, not the MP10 Mini.
- Your current firmware is official full-size v72 (`M115` reports `VERSION: 72`).
- Your stock firmware reports full-size geometry, normally `M211` max `X300 Y300 Z400`.
- Your machine cannot complete stock high-Y calibration/probing without hitting the front Y limit.
- You can supervise the firmware flash and first post-flash calibration.

Do not use this on an MP10 Mini, unknown firmware, or a printer whose stock Y travel safely clears the normal MP10 calibration envelope.

## Rebuild The Patched Image

Requires Python 3 and no third-party packages.

```bash
python3 tools/create_update_bin.py \
  --source /path/to/official/full-size/v72/update.bin \
  --output-dir firmware \
  --force
sha256sum -c firmware/SHA256SUMS.txt
```

The script refuses to run unless the source image hash matches the official full-size MP10 v72 image above.

The generator writes both `firmware/custom_update.bin` and `firmware/update.bin`.
They are byte-for-byte identical and have the same SHA-256:

```text
51e61467e79c5cfd9f59300a8665577da1bed0d14209e6dc717ff7c3f374a7f5
```

The distinction is naming only:

- `custom_update.bin` is the CI/build/release artifact so it is obvious this is not a vendor file.
- `update.bin` is the required on-card name for the MP10/Malyan bootloader.

CI verifies this by rebuilding from the official v72 source image, comparing both generated files to the committed firmware files, checking that `custom_update.bin` and `update.bin` are byte-identical, and validating `firmware/SHA256SUMS.txt`.

## Release Automation

GitHub Actions runs firmware verification on pull requests, pushes to `main`, and release tags. Tag pushes matching `v*` or `firmware-*` also publish a GitHub Release containing:

- `custom_update.bin`
- `update.bin`
- `manifest.json`
- `manifest.md`
- `SHA256SUMS.txt`

The release job rebuilds these assets from the official full-size MP10 v72 image before upload. It does not upload stale workspace files.

## Regenerate Binary Analysis

Requires Python 3 and Ghidra `analyzeHeadless`.

```bash
python3 tools/decompile_official_v72.py \
  --ghidra /path/to/ghidra/support/analyzeHeadless \
  --source /path/to/official/full-size/v72/update.bin \
  --output-dir src/official-v72 \
  --force
```

The generated output is useful for auditing the binary patch locations. It is not a substitute for vendor source.

## Flash

1. Copy [firmware/update.bin](firmware/update.bin) to the root of a FAT-formatted microSD card as `update.bin`.
   - If you downloaded `custom_update.bin` from a GitHub Release, rename it to `update.bin` on the card before flashing.
2. Power the printer off.
3. Insert the card and power the printer on.
4. Wait for the firmware update process to finish.
5. Remove or rename `update.bin` on the card after flashing so it does not flash repeatedly.

Keep official v72 rollback media available.

## Verify Before Motion

After flashing, connect over USB and run read-only checks before heating, extrusion, probing, or printing:

```gcode
M115
M211
M503
G29 W
M420 V
M119
M105
M27
M114
```

Expected key evidence:

- `M115` still reports `VERSION: 72`.
- `M211` reports max `Y260.00`.
- `G29 W` reports Y-axis mesh points ending at `260.000`.
- Supervised `G29 P1` does not probe above `Y260`.

Known display caveat: `M420 V` topography text can still print stale corner labels like `(20,290)`. Use `M211`, `G29 W`, `G29 S-1`, and a supervised verbose probe path as the motion evidence.
